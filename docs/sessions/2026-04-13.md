## 핵심 설계 결정

### 보안 구조
- 모든 서버사이드 코드(`api/`, `app/api/`) → `SUPABASE_SERVICE_ROLE_KEY` (RLS 우회)
- 브라우저 → Next.js API Route → Supabase 구조 (브라우저가 Supabase 직접 접근 없음)
- 승인 페이지 인증: Form POST → `approve_token` 쿠키에 비밀번호 저장 → 이후 요청마다 `Authorization: Bearer {password}` 헤더로 Python 백엔드 검증

### 타임슬롯 전략 (Q1+Q2 Option A)
| UTC | KST | 슬롯 | 스타일 |
|-----|-----|------|--------|
| 23:10 (전날) | 08:10 | KR | 한국어 불렛 포인트형 |
| 01:25 | 10:25 | KR | 동일 |
| 03:40 | 12:40 | KR | 동일 |
| 06:00 | 15:00 | KR | 동일 |
| 08:10 | 17:10 | KR | 동일 |
| **13:30** | **22:30** | **US** | **영어 인사이트형 (DC 타겟)** |

### DB 상태 흐름
```
pending → posted    (승인 + 트윗 성공)
pending → rejected  (거절)
pending → failed    (트윗 실패, error_message 기록)
```

## 다음 세션: 수동 세팅 및 배포

### 외부 서비스 세팅 순서
1. **Supabase** — 프로젝트 생성 + `001_init.sql` 실행 (10분)
2. **Anthropic** — API Key 발급 (5분)
3. **X API** — @ANDCenter_NK 앱 생성, Read+Write 권한 설정 (승인 수일 소요 가능)
4. **Slack** — Incoming Webhook URL 생성 (5분)
5. **비밀번호** — 승인 페이지용 비밀번호 결정

### 로컬 테스트
```bash
npm install
pip install -r requirements.txt
cp .env.example .env.local  # 값 채우기
npx vercel dev

# 파이프라인 테스트
curl http://localhost:3000/api/cron

# 승인 페이지 테스트
open http://localhost:3000/approve/{supabase에서 생성된 draft_id}
```

### Vercel 배포
```bash
npx vercel --prod
```
Vercel Dashboard → Settings → Environment Variables에 `.env.example` 키 10개 입력 후 재배포
